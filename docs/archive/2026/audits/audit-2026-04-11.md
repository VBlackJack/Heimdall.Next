## Executive Summary

Heimdall.Next est un projet solide avec une architecture claire, une sécurité défensive bien pensée (DPAPI, InputValidator, TOFU, shell escaping), et une couverture de tests correcte pour le domaine Core. **La cause racine du bug SSH password auth est identifiée** : `SshConnectionFactory.BuildAuthMethods()` n'ajoute jamais `KeyboardInteractiveAuthenticationMethod`, ce qui bloque les connexions sur tous les serveurs qui n'acceptent pas le type `password` directement (cas courant sur les Linux durcis). Le code anticipe d'ailleurs ce cas (`SshFailureCode.KeyboardInteractiveNoPassword` existe) mais l'implémentation n'a jamais suivi. Les autres findings sont limités et bien traçables.

**Total findings : 1 🔴 Critique · 5 🟠 Important · 3 🟡 Mineur**

---

## Findings by Severity

### 🔴 Critical (1)

**SSH-01 — `KeyboardInteractiveAuthenticationMethod` manquant dans le path SSH.NET.**
Quand un serveur SSH n'accepte que `keyboard-interactive` (et pas `password` en tant que méthode SSH), SSH.NET échoue silencieusement. L'auth Plink (Pageant/certificat) fonctionne car Plink gère `keyboard-interactive` nativement. C'est exactement le symptôme décrit.

### 🟠 Important (5)

**SSH-02 — Fallback Plink conditionnel à Pageant disponible.**
Quand SSH.NET échoue et Pageant n'est pas actif, aucun fallback n'est tenté : l'utilisateur reçoit une erreur sans recours même si Plink pourrait réussir avec le mot de passe.

**SSH-03 — `Create()` et `CreateForTunnel()` sont identiques.**
Les deux méthodes ont une implémentation copiée-collée. Un correctif dans l'une ne propagera pas à l'autre.

**SSH-04 — `SshShellSession.Resize()` repose sur de la réflexion privée.**
Accès au champ `_channel` de SSH.NET par `BindingFlags.NonPublic`. Une mise à jour de SSH.NET casse silencieusement le resize (loggué en `Warn`, transparent pour l'utilisateur).

**TEST-01 — Aucun test pour `SshConnectionFactory` et `SshShellSession`.**
La composition des méthodes d'auth (code critique, source du bug) n'est couverte par aucun test unitaire. Le bug aurait été attrapé par un test `BuildAuthMethods_PasswordOnly_ShouldIncludeKeyboardInteractive`.

**CI-01 — `dotnet format` en `continue-on-error: true`.**
Les violations de formatage n'échouent pas le build CI. La dérive de style peut s'accumuler sans blocage.

### 🟡 Minor (3)

**CQ-01 — `ReadBufferSize` devrait être `const`.**
`private static readonly int ReadBufferSize = 8192;` → `private const int ReadBufferSize = 8192;`. Valeur connue à la compilation.

**UX-01 — Fallback Plink post-échec SSH.NET ne transmet pas le mot de passe.**
Quand SSH.NET échoue et Plink prend le relais, aucun `-pw` n'est passé. L'utilisateur doit retaper son mot de passe dans le terminal xterm.js. Fonctionnel mais sub-optimal.

**CQ-02 — `SshFailureCode.KeyboardInteractiveNoPassword` jamais émis.**
Le code existe dans l'enum et dans la clé i18n attendue (`ErrorSshKeyboardInteractiveNoPassword`), mais `FailureClassifier` ne le produit jamais. Dead code partiel.

## 🎯 Root Cause : Bug SSH password auth — Analyse complète

### Pourquoi ça bug avec login/password mais pas avec certificat/Pageant

Il y a deux chemins de connexion SSH dans Heimdall.Next :

**Chemin A — SSH.NET** (utilisé pour password + clé sans Pageant)
```
ConnectSshAsync → SshConnectionFactory.Create() → SshShellSession.ConnectAsync()
→ SSH.NET ShellStream → DataReceived events → xterm.js
```

**Chemin B — Plink pipe mode** (utilisé pour Pageant / agent forwarding)
```
ConnectSshAsync → RequiresPageantFallback() = true → ConnectSshViaPlinkAsync()
→ PipeModeSession → stdin/stdout pipe → xterm.js
```

### Le bug

`SshConnectionFactory.BuildAuthMethods()` construit la liste des méthodes d'auth SSH.NET :

```csharp
// Ce qui est fait — seul PasswordAuthenticationMethod est ajouté
if (!string.IsNullOrEmpty(connectionParams.Password) &&
    string.IsNullOrWhiteSpace(connectionParams.KeyPath))
{
    methods.Add(new PasswordAuthenticationMethod(connectionParams.Username, connectionParams.Password));
    // ← MANQUANT : KeyboardInteractiveAuthenticationMethod
}
```

`PasswordAuthenticationMethod` envoie le type d'auth SSH `"password"` (RFC 4252 §8). Beaucoup de serveurs SSH modernes ont :
```
PasswordAuthentication no          # désactive le type "password"
KbdInteractiveAuthentication yes   # accepte "keyboard-interactive"
```

Dans ce cas, SSH.NET reçoit `SSH_MSG_USERAUTH_FAILURE` et lève `SshAuthenticationException: "No supported authentication method found"`.

Plink, lui, essaie automatiquement `keyboard-interactive`, qui est le vrai mécanisme de challenge-réponse implémenté côté serveur — et ça marche.

**La preuve que c'était anticipé :** `SshFailureCode.KeyboardInteractiveNoPassword` est défini dans `SshFailureCode.cs:63` mais jamais émis par `FailureClassifier`. Le code failure était prévu, l'auth method ne l'a jamais été.

### Fix — 3 lignes dans `SshConnectionFactory.BuildAuthMethods()`

```csharp
// Password authentication
if (!string.IsNullOrEmpty(connectionParams.Password) &&
    string.IsNullOrWhiteSpace(connectionParams.KeyPath))
{
    // Type SSH "password" — fonctionne si le serveur accepte PasswordAuthentication
    methods.Add(new PasswordAuthenticationMethod(connectionParams.Username, connectionParams.Password));

    // Type SSH "keyboard-interactive" — requis sur les serveurs avec PasswordAuthentication no
    // SSH.NET essaie les méthodes dans l'ordre : password d'abord, kbdint si password refusé
    var password = connectionParams.Password; // closure capture
    var kbdInteractive = new KeyboardInteractiveAuthenticationMethod(connectionParams.Username);
    kbdInteractive.AuthenticationPrompt += (_, e) =>
    {
        foreach (var prompt in e.Prompts)
        {
            prompt.Response = password;
        }
    };
    methods.Add(kbdInteractive);
}
```

Et émettre `KeyboardInteractiveNoPassword` dans `FailureClassifier` si le mot de passe est vide lors d'une tentative kbdint.

## What Was NOT Flagged (and why)

- **`SshShellSession.ReadAsync` retournant 0** — Vérifié : SSH.NET 2025.1.0 `ShellStream.ReadAsync` retourne 0 uniquement à l'EOF réel. Pas de faux 0 sur cette version. Le `break` sur `bytesRead <= 0` est correct.
- **Password Plink sans `-pw`** — Fonctionnel : Plink affiche son prompt via le pipe stdout/stderr, xterm.js le montre, l'utilisateur tape. Ce n'est pas un bug de sécurité (le mot de passe ne circule pas en clair sur le réseau ou en mémoire de façon dangereuse), juste une UX dégradée (pas de masking dans la fenêtre Heimdall avant connexion).
- **`PageantKeyWrapper` + `PrivateKeyAuthenticationMethod` toujours ajouté** — Design correct : le fallback Pageant côté SSH.NET est supplémentaire. Si password réussit, SSH.NET s'arrête. Pas de risque d'épuisement de `MaxAuthTries` car password est tenté en premier.
- **`ProbeHostKeyFingerprintAsync` sans cleanup du process** — Le `process.Kill(true)` sur timeout est correct ; pas de zombie possible.
- **MVVM dans `EmbeddedSshView.xaml.cs`** — Code-behind WPF lourd mais légitime pour ce type de vue (WebView2 lifecycle, focus management, event wiring). Pas de logique métier dans le code-behind.
- **Aucun test de connexion SSH réel** — Correct pour des tests unitaires. Les tests SSH se font via mocks SSH.NET. Les tests d'intégration vrais nécessiteraient un serveur SSH embarqué (acceptable de ne pas avoir).
