# Project Audit Report — Heimdall.Next

**Date:** 2026-04-22
**Stack:** .NET 10 (C# 14) + WPF MVVM, CommunityToolkit.Mvvm, SSH.NET, WebView2, EF Core + SQLite (TwinShell), LibGit2Sharp
**Mode:** Full audit (10 categories)
**Scope:** Entire repo — 9 src projects + 4 test projects, 2183 .cs files, 99 .xaml files, 287 test files, config/, locales/, CI, build scripts
**Build under audit:** 1.0.0421.2 / InformationalVersion 2026.042102
**Auditor:** project-audit skill — Explore subagents x5 (Code+Arch, Security, Performance, UX+UI+A11Y, Doc+Test+DevOps)

---

## Executive Summary

Heimdall.Next is a mature, well-engineered codebase. No **critical** findings were identified in any of the ten categories. The architecture is sound (clean dependency graph, MVVM respected, DPAPI/HMAC crypto correctly implemented with constant-time comparison, `.editorconfig` + `TreatWarningsAsErrors` + `packages.lock.json` across all projects), CI enforces format, build, test, vuln check and i18n key parity, and the 4195-test baseline covers security, state machine, tunnels, SSH, split service and most ViewModels.

The audit surfaced **21 Important** and ~15 **Minor** findings. The themes are recurring: (1) a handful of silent `async void` / swallowed exception paths that will hide production failures, (2) two concrete event-handler leaks in WebView2-backed views that matter in long-running sessions, (3) one UI-thread blocking `GetAwaiter().GetResult()` on startup, (4) a small cluster of input-validation gaps on `Process.Start` arguments (nslookup, PuTTY `-i` path) and one network-exposed HTTP server (`EphemeralFileServer` binds `http://+`), (5) a few MVVM leaks (WPF types in ViewModels / Models), (6) drift between `README.md`, `CLAUDE.md` and the actual tool / locale-key counts. Nothing here blocks release, but the event-handler leaks, the blocking startup call, and the `EphemeralFileServer` network exposure are the three that genuinely hurt user experience or security posture and should ship in the next patch.

**SEC-I1 (Important) — `nslookup` argument injection**
`Services/DnsLookupService.cs:209–211` — `Arguments = $"-type={recordType} {hostname} {dnsServer}"` with no quoting/escaping. `UseShellExecute=false` prevents shell injection, but nslookup's own arg parser accepts extra flags. The ViewModel validates hostname via `InputValidator.ValidateDomain`, but the delegate is injectable and has no internal guard. `dnsServer` (from user-configurable presets) is completely unescaped. Fix: validate inside `LocalNslookupQueryAsync`, double-quote the hostname.

**SEC-I2 (Important) — SSH key path double-quote escape**
`Services/Handlers/SshHandler.cs:215`, `Heimdall.Ssh/Plink/PlinkTunnelRunner.cs:264` — `$"\"{server.SshKeyPath}\""` breaks if the path itself contains a `"`. Paths via `OpenFileDialog` are safe, but paths imported from `.rdg` files or pasted are not. Fix: escape embedded quotes or validate the path pattern.

**SEC-I3 (Important) — `EphemeralFileServer` binds on all interfaces with no auth**
`Services/EphemeralFileServer.cs:81` — `http://+:{port}/`. Any machine on the same LAN can enumerate and download files for the duration the server is active. The warning at line 86–89 is log-only, not UI-visible. Fix: (a) surface a warning in the FileShare tool UI when the server is active on public interfaces, (b) add a per-session token generated at start and shown in the UI so only the user who launched the share can fetch.

**SEC-I4 (Important) — WebView2 referenced as local DLL, version opaque**
`Heimdall.App.csproj:25–30` — shipped from `lib/webview2/*.dll` with no NuGet pin. `dotnet list package --vulnerable` cannot audit it; Dependabot/Renovate won't flag advisories. Evergreen runtime auto-updates on the user's machine, but the SDK bridge is frozen. Fix: migrate to `<PackageReference Include="Microsoft.Web.WebView2" Version="..." />`.

**Minor SEC findings**
- **SEC-M1** `PrivilegeLauncher.LaunchViaSelfElevation` appends user `args` unsanitized, self-elevation child only — low risk (`Heimdall.Core/Security/PrivilegeLauncher.cs:347–349`).
- **SEC-M2** `WebSocketVncProxy` accepts any local WebSocket client once Origin passes — local-only surface (`Services/WebSocketVncProxy.cs`).
- **SEC-M3** `LocalFileBrowserView` passes `entry.FullPath` to `rundll32`/`explorer`/editor with naive quoting — paths with embedded `"` would break arg parsing (`Views/LocalFileBrowserView.xaml.cs:268,289,353`).

**PERF-I1 (Important) — UI-thread blocking `GetAwaiter().GetResult()` at startup**
`App.xaml.cs:376` — `var settings = configManager.LoadSettingsAsync().GetAwaiter().GetResult();` is executed from a DI factory delegate during `services.BuildServiceProvider()`, synchronously on the UI thread inside `OnStartup`. File I/O here = unbounded UI freeze if the settings file is on a slow drive or AV-scanned. Fix: pre-load `AppSettings` from the async flow already present at line 121 and pass the resolved path to the factory.

**PERF-I2 (Important) — `NavigationStarting` lambda leak in `EmbeddedVncView`**
`Views/EmbeddedVncView.xaml.cs:153` — anonymous `core.NavigationStarting += (_, navArgs) => { ... }` never removed in `Dispose`. `CoreWebView2` can outlive the view (environment cache). One delegate per session in a long-running app.

**PERF-I3 (Important) — Same pattern in `EmbeddedSshView`**
`Views/EmbeddedSshView.xaml.cs:640` — same anonymous `NavigationStarting` handler, same Dispose gap.

**PERF-I4 (Important) — `EditInEditorRequested` event subscription not removed**
`Services/EmbeddedSessionManager.cs:169` — `fileBrowser.EditInEditorRequested += async (path) => {...}` captured lambda never removed. In a heavy-use local-shell tab, each editor open stacks a handler. Also nests an inner `editorView.CloseRequested += () => {...}` that has no removal on editor close.

**PERF-I5 (Important) — Terminal HTML + assets re-read from disk per session**
`Views/EmbeddedSshView.xaml.cs:1407–1475` — `GetTerminalHtml()` calls `ReadTerminalAsset("terminal.html")` + 4 more asset reads synchronously on the UI thread in `Loaded`. Each SSH tab/split pane re-reads ~300–500 KB. Fix: `static readonly` cached strings populated lazily once.

**PERF-I6 (Important) — `async void` non-event methods in `SidebarViewModel`**
`ViewModels/Sidebar/SidebarViewModel.cs:295,305` — `PersistTabChoice` and `OnCategoryPropertyChanged` are `private async void` but not event handlers. Exceptions go to `TaskScheduler.UnobservedTaskException`, invisible to users. Fix: `async Task` + explicit error handling at call site.

**PERF-I7 (Important) — `Task.Run(async () => InitializeAsync(...))` → `Task<Task>`**
`App.xaml.cs:138` — `await Task.Run(() => TwinShellBootstrapper.InitializeAsync(_serviceProvider));` — the outer `await` awaits the outer `Task.Run` (scheduling), not the inner async task. Verify whether `InitializeAsync` is actually sync internally (wrapping sync DB seed ops) or whether the form should be `await Task.Run(async () => await TwinShellBootstrapper.InitializeAsync(_serviceProvider))` or simply `await TwinShellBootstrapper.InitializeAsync(_serviceProvider)`.

**PERF-I8 (Important) — Command library history `ListView` virtualization**
`Views/Tools/CommandLibraryView.xaml:553` — no explicit `VirtualizingPanel.*` attributes and a fully custom `ControlTemplate` on `ListViewItem`. Inherits through default style in `CommonControls.xaml` (which does set `IsVirtualizing=True`), so it is virtualized — but a brittle link. Add explicit attributes on both the main action list and the history list for robustness.

**PERF-I9 (Important) — WebSocketVncProxy 64 KB buffer allocated per receive**
`Services/WebSocketVncProxy.cs:199,226` — `var buffer = new byte[BufferSize];` in two tight relay loops. Under an active VNC session (screen refresh traffic) this generates continuous GC pressure. Fix: `ArrayPool<byte>.Shared.Rent(BufferSize)` + `Return` in `finally`.

**Minor PERF findings**
- `SidebarViewModel.OnCategoryPropertyChanged` uses synchronous `Dispatcher.Invoke` where `InvokeAsync` would be less risky.
- `TwinShell.Infrastructure/Services/HealthCheckService.cs:270` creates `new HttpClient()` per call (disposed via `using`, but bypasses connection pooling).
- `ThemeService.Resolve` does an O(7) `.First(k.Equals(...))` — cold path, acceptable.
